🔍 1. What is the usage?

Compute Engine is used to run virtual machines (VMs) in Google's data centers. It provides Infrastructure as a Service (IaaS) capabilities, allowing you to create and manage virtual servers with full control over operating systems, configurations, and resources. Common use cases include:

  • Running web applications and APIs
  • Hosting databases and data processing workloads
  • Development and testing environments
  • High-performance computing tasks
  • Legacy application migration to the cloud
  • Batch processing and analytics workloads

⏰ 2. When do we use?

Use Compute Engine when you need:

  • Full control over your infrastructure and operating systems
  • Custom hardware configurations (CPU, memory, storage)
  • Long-running applications that can't tolerate cold starts
  • Applications requiring specific OS versions or kernel configurations
  • Workloads needing direct access to GPUs or specialized hardware
  • Legacy applications that require traditional server architectures
  • High-performance computing requiring low-latency networking

🏗️ 3. Architecture of the tool?

Compute Engine's architecture consists of:

  • Virtual Machines: KVM-based virtualization running on physical servers
  • Persistent Disks: Network-attached storage with automatic encryption
  • VPC Networks: Software-defined networking with global reach
  • Regions and Zones: Geographic distribution for high availability
  • Live Migration: Zero-downtime maintenance and upgrades
  • Metadata Server: Instance metadata and configuration service

⚙️ 4. How to configure?

To configure Compute Engine instances:

  1. Create a VM instance: Use gcloud CLI, Cloud Console, or API
  2. Choose machine type: Select appropriate CPU/memory combination
  3. Configure disks: Attach boot disk and additional persistent disks
  4. Set up networking: Configure VPC, subnets, and external IP
  5. Configure security: Set up service accounts and IAM roles
  6. Add metadata: Configure startup scripts and custom metadata
  7. Set up monitoring: Enable Cloud Monitoring and logging
Basic VM Configuration
# Create a basic VM instance
gcloud compute instances create my-instance \
  --zone=us-central1-a \

python
from google.cloud import compute_v1

def create_instance(project_id, zone, instance_name):
    """Create a basic Compute Engine instance."""
    client = compute_v1.InstancesClient()

    instance = compute_v1.Instance()
    instance.name = instance_name
    instance.machine_type = f"zones/{zone}/machineTypes/n1-standard-1"

    # Boot disk configuration
    boot_disk = compute_v1.AttachedDisk()
    boot_disk.auto_delete = True
    boot_disk.boot = True
    boot_disk.initialize_params = compute_v1.AttachedDiskInitializeParams()
    boot_disk.initialize_params.source_image = "projects/debian-cloud/global/images/family/debian-11"
    instance.disks = [boot_disk]

    # Network interface
    network_interface = compute_v1.NetworkInterface()
    network_interface.name = "global/networks/default"
    instance.network_interfaces = [network_interface]

    operation = client.insert(
        project=project_id,
        zone=zone,
        instance_resource=instance
    )

    print(f"Creating instance {instance_name}...")
    operation.result()
    print(f"Instance {instance_name} created successfully!")

🔍 1. What is the usage?

Regions and zones in Google Cloud Compute Engine represent the physical and logical infrastructure where your VMs run. Regions are geographic locations containing multiple zones, while zones are isolated locations within a region. This global infrastructure enables:

  • Global Distribution: Deploy applications worldwide for better user experience
  • High Availability: Spread workloads across zones to survive zone failures

🎯 Scaling Strategy Decision Framework

1. Workload Characteristics

  • Batch Processing: Use preemptible VMs with job queues

2. Availability Requirements

  • High Availability: Regional MIGs + global load balancing
  • Cost Optimization: Zonal MIGs + preemptible instances
  • Disaster Recovery: Multi-region deployment with cross-region load balancing

3. Performance Considerations

  • Low Latency: Regional load balancing with proximity-based routing
  • High Throughput: TCP/UDP load balancing for non-HTTP traffic
  • Global Reach: Global HTTP(S) load balancing with CDN integration

Firewall Rules & Network Security

Intermediate

VPC firewall rules control traffic to and from VM instances. They provide fine-grained network security by allowing or denying traffic based on source, destination, protocol, and ports.

Firewall Rule Management
# Allow HTTP traffic to web servers
gcloud compute firewall-rules create allow-http \
    --allow=tcp:80 \
    --target-tags=web-server \
    --source-ranges=0.0.0.0/0 \
    --description="Allow HTTP traffic from anywhere"

# Allow SSH from specific IP range
gcloud compute firewall-rules create allow-ssh-internal \
    --allow=tcp:22 \
    --source-ranges=192.168.1.0/24 \
    --description="Allow SSH from internal network"

# Deny all traffic from specific IP
gcloud compute firewall-rules create deny-bad-ip \
    --action=DENY \
    --rules=all \
    --source-ranges=203.0.113.0/24 \
    --priority=100

# Allow traffic between instances with same tag
gcloud compute firewall-rules create allow-internal \
    --allow=tcp,udp,icmp \
    --source-tags=web-server \
    --target-tags=database

# Allow health check traffic
gcloud compute firewall-rules create allow-health-checks \
    --allow=tcp:80 \
    --source-ranges=130.211.0.0/22,35.191.0.0/16 \
    --target-tags=web-server

# Service account based rules
gcloud compute firewall-rules create allow-service-account \
    --allow=tcp:443 \
    --source-service-accounts=my-app@my-project.iam.gserviceaccount.com \
    --target-tags=api-server

# Create instance with network tags
gcloud compute instances create tagged-instance \
    --tags=web-server,allow-ssh \
    --zone=us-central1-a

# Add tags to existing instance
gcloud compute instances add-tags existing-instance \
    --tags=database,internal-only \
    --zone=us-central1-a

# Remove specific tags
gcloud compute instances remove-tags instance-name \
    --tags=old-tag \
    --zone=us-central1-a

# List instances by tag
gcloud compute instances list --filter="tags:web-server"
🛡️ Firewall Security Best Practices
  • Default Deny: VPC firewall denies all traffic by default
  • Network Tags: Use tags to apply rules to groups of instances
  • Priority Order: Lower numbers have higher priority
  • Regular Audits: Review and clean up unused firewall rules

# Discount percentage based on usage hours

# 50-75% usage: 25% discount
# 75-100% usage: 30% discount

# Monthly hours = 30 * 24 = 720
# Usage hours = instance running time
# Usage percentage = (usage hours / 720) * 100

  TARGET_SIZE=10
