IAM Overview

Basic

Identity and Access Management (IAM) enables you to manage access control by defining who (identity) has what access (role) for which resource. IAM provides fine-grained access control and is a critical component of Google Cloud security.

🔐 Key Principles
  • Least Privilege: Grant only necessary permissions
  • Separation of Duties: Different roles for different tasks
  • Defense in Depth: Multiple layers of access control
  • Audit Everything: Comprehensive logging and monitoring

Principals & Identity Types

Basic

Principals are identities that can be granted access to resources. GCP supports various types of principals including Google accounts, service accounts, and groups.

👤
Google Accounts
Individual user accounts with @gmail.com or custom domains
🤖
Service Accounts
Accounts for applications and services to authenticate
👥
Google Groups
Collections of users for simplified access management
🌐
Domain Users
Users from Google Workspace or Cloud Identity domains
1# List all IAM principals in a project
2gcloud projects get-iam-policy my-project --format="table(bindings.role,bindings.members)"
3
4# Add user to project with specific role
5gcloud projects add-iam-policy-binding my-project \
6 --member="user:alice@example.com" \
7 --role="roles/viewer"
8
9# Add service account to project
10gcloud projects add-iam-policy-binding my-project \
11 --member="serviceAccount:my-service@my-project.iam.gserviceaccount.com" \
12 --role="roles/editor"
13
14# Add Google Group to project
15gcloud projects add-iam-policy-binding my-project \
16 --member="group:developers@mycompany.com" \
17 --role="roles/cloudbuild.builds.editor"
18
19# Create service account
20gcloud iam service-accounts create my-service-account \
21 --description="My service account" \
22 --display-name="My Service Account"
23
24# List service accounts
25gcloud iam service-accounts list
26
27# Create service account key
28gcloud iam service-accounts keys create key.json \
29 --iam-account=my-service-account@my-project.iam.gserviceaccount.com

Roles & Permission Hierarchy

Intermediate

Roles are collections of permissions that can be granted to principals. GCP provides basic roles, predefined roles, and allows creation of custom roles for specific needs.

1# List predefined roles
2gcloud iam roles list --filter="name:*compute*"
3
4# Describe a specific role
5gcloud iam roles describe roles/compute.instanceAdmin
6
7# Create custom role
8gcloud iam roles create my-custom-role \
9 --project=my-project \
10 --title="My Custom Role" \
11 --description="Custom role for specific permissions" \
12 --permissions="compute.instances.get,compute.instances.list,compute.disks.get"
13
14# Update custom role
15gcloud iam roles update my-custom-role \
16 --project=my-project \
17 --add-permissions="compute.instances.start"
18
19# List custom roles
20gcloud iam roles list --filter="name:*my-custom*" --project=my-project
21
22# Delete custom role
23gcloud iam roles delete my-custom-role --project=my-project
24
25# Test IAM permissions
26gcloud iam permissions test-iam-permissions \
27 --member="user:alice@example.com" \
28 --resource="//cloudresourcemanager.googleapis.com/projects/my-project" \
29 --permissions="compute.instances.create,compute.disks.create"
🎭 Role Types
  • Basic Roles: Owner, Editor, Viewer (broad permissions)
  • Predefined Roles: Service-specific roles (e.g., compute.admin)
  • Custom Roles: User-defined combinations of permissions
  • Primitive Roles: Legacy roles (avoid for new projects)

Service Accounts & Authentication

Intermediate

Service accounts are special accounts for applications and services to authenticate with Google Cloud APIs. They provide secure, key-based authentication without user interaction.

1# Create service account
2gcloud iam service-accounts create my-app-service \
3 --description="Service account for my application" \
4 --display-name="My App Service Account"
5
6# Create service account key
7gcloud iam service-accounts keys create ~/key.json \
8 --iam-account=my-app-service@my-project.iam.gserviceaccount.com
9
10# List service account keys
11gcloud iam service-accounts keys list \
12 --iam-account=my-app-service@my-project.iam.gserviceaccount.com
13
14# Delete old service account key
15gcloud iam service-accounts keys delete KEY_ID \
16 --iam-account=my-app-service@my-project.iam.gserviceaccount.com
17
18# Grant roles to service account
19gcloud projects add-iam-policy-binding my-project \
20 --member="serviceAccount:my-app-service@my-project.iam.gserviceaccount.com" \
21 --role="roles/storage.objectViewer"
22
23# Use service account impersonation
24gcloud auth activate-service-account my-app-service@my-project.iam.gserviceaccount.com \
25 --key-file=~/key.json
26
27# Create service account with limited permissions
28gcloud iam service-accounts create restricted-service \
29 --description="Restricted service account" \
30 --display-name="Restricted Service"
31
32# Disable service account
33gcloud iam service-accounts disable my-app-service@my-project.iam.gserviceaccount.com
34
35# Delete service account
36gcloud iam service-accounts delete my-app-service@my-project.iam.gserviceaccount.com
🤖 Service Account Best Practices
  • Principle of Least Privilege: Grant minimal required permissions
  • Key Rotation: Regularly rotate service account keys
  • Separate Accounts: Use different accounts for different services
  • Workload Identity: Use workload identity instead of keys when possible
  • Monitoring: Monitor service account usage and anomalies

IAM Policies & Bindings

Intermediate

IAM policies define who has what access to which resources. Policies are attached to resources and contain bindings that associate principals with roles. Understanding policy structure is crucial for effective access management.

1# Create policy binding
2gcloud projects add-iam-policy-binding my-project \
3 --member="user:bob@example.com" \
4 --role="roles/compute.instanceAdmin.v1"
5
6# Remove policy binding
7gcloud projects remove-iam-policy-binding my-project \
8 --member="user:bob@example.com" \
9 --role="roles/compute.instanceAdmin.v1"
10
11# Get IAM policy
12gcloud projects get-iam-policy my-project --format="table(bindings.role,bindings.members)"
13
14# Set IAM policy from file
15gcloud projects set-iam-policy my-project policy.json
16
17# Check effective permissions
18gcloud iam permissions test-iam-permissions \
19 --member="user:alice@example.com" \
20 --resource="//cloudresourcemanager.googleapis.com/projects/my-project" \
21 --permissions="compute.instances.create,compute.disks.create"
22
23# List service account keys
24gcloud iam service-accounts keys list \
25 --iam-account=my-service-account@my-project.iam.gserviceaccount.com
26
27# Create service account key
28gcloud iam service-accounts keys create key.json \
29 --iam-account=my-service-account@my-project.iam.gserviceaccount.com
30
31# Delete service account key
32gcloud iam service-accounts keys delete KEY_ID \
33 --iam-account=my-service-account@my-project.iam.gserviceaccount.com
📄 Policy Structure
  • Bindings: Associate members with roles
  • Members: Principals (users, groups, service accounts)
  • Roles: Collections of permissions
  • Conditions: Optional constraints on when binding applies
  • Version: Policy version (1 or 2 for conditions)
