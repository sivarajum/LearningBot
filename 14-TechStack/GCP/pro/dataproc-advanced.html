Cloud Storage Integration

Intermediate

Dataproc clusters can directly read from and write to Cloud Storage buckets without copying data to HDFS. This provides seamless integration and eliminates data duplication.

PySpark: Read from GCS and Process Data
from pyspark.sql import SparkSession # Create Spark session with GCS connector spark = SparkSession.builder \ .appName("GCS-Dataproc-Integration") \ .config("spark.hadoop.fs.gs.impl", "com.google.cloud.hadoop.fs.gcs.GoogleHadoopFileSystem") \ .config("spark.hadoop.google.cloud.auth.service.account.enable", "true") \ .getOrCreate() # Read CSV from Cloud Storage input_path = "gs://your-bucket/input-data/*.csv" df = spark.read.csv(input_path, header=True, inferSchema=True) # Data transformations processed_df = df \ .filter(df["status"] == "active") \ .groupBy("category") \ .agg({"revenue": "sum", "orders": "count"}) \ .withColumnRenamed("sum(revenue)", "total_revenue") \ .withColumnRenamed("count(orders)", "order_count") # Write results back to Cloud Storage output_path = "gs://your-bucket/processed-data/results.parquet" processed_df.write.parquet(output_path, mode="overwrite") print(f"Processed {processed_df.count()} records")
⚡ GCS Integration Benefits
  • No Data Copying: Direct access to GCS objects
  • Cost Effective: No HDFS storage costs for data
  • Scalable: Unlimited storage capacity
  • Reliable: 99.999999999% durability

BigQuery Integration

Advanced

Dataproc integrates seamlessly with BigQuery for advanced analytics workflows. Process large datasets with Spark and load results directly into BigQuery for analysis.

Spark: Process Data and Load to BigQuery
from pyspark.sql import SparkSession from google.cloud import bigquery # Initialize Spark with BigQuery connector spark = SparkSession.builder \ .appName("Dataproc-BigQuery-Integration") \ .config("spark.jars.packages", "com.google.cloud.spark:spark-bigquery-with-dependencies_2.12:0.23.2") \ .getOrCreate() # Read large dataset from Cloud Storage raw_data = spark.read.parquet("gs://data-lake/raw-events/*.parquet") # Complex data processing with Spark processed_data = raw_data \ .filter("event_type IN ('purchase', 'signup', 'login')") \ .withColumn("event_date", F.date_format("event_timestamp", "yyyy-MM-dd")) \ .groupBy("user_id", "event_date") \ .agg( F.count("*").alias("event_count"), F.sum("revenue").alias("total_revenue"), F.max("event_timestamp").alias("last_activity") ) \ .withColumn("user_segment", F.when(F.col("total_revenue") > 1000, "high_value") .when(F.col("total_revenue") > 100, "medium_value") .otherwise("low_value") ) # Load processed data directly to BigQuery processed_data.write \ .format("bigquery") \ .option("table", "project.dataset.user_analytics") \ .option("temporaryGcsBucket", "temp-bucket") \ .mode("overwrite") \ .save() print(f"Loaded {processed_data.count()} user analytics records to BigQuery")
🔄 Complete Data Pipeline
  • Step 1: Raw data lands in Cloud Storage
  • Step 2: Dataproc processes data with Spark
  • Step 3: Results loaded directly to BigQuery
  • Step 4: Analytics and reporting in BigQuery

Intelligent Autoscaling

Advanced

Dataproc's autoscaling automatically adjusts cluster size based on workload demands, optimizing both performance and cost. Scale up during peak processing and scale down during idle periods.

gcloud: Create Autoscaling Cluster
# Create cluster with autoscaling enabled gcloud dataproc clusters create my-autoscaling-cluster \ --region=us-central1 \ --master-machine-type=n1-standard-4 \ --master-boot-disk-size=500 \ --num-workers=2 \ --worker-machine-type=n1-standard-4 \ --worker-boot-disk-size=500 \ --image-version=2.0-debian10 \ --enable-component-gateway \ --optional-components=ANACONDA,JUPYTER \ --max-idle=30m \ --autoscaling-policy=my-autoscaling-policy # Create autoscaling policy gcloud dataproc autoscaling-policies import my-autoscaling-policy \ --source=autoscaling-policy.yaml \ --region=us-central1 # Autoscaling policy configuration (autoscaling-policy.yaml) workerConfig: minInstances: 2 maxInstances: 100 weight: 1 secondaryWorkerConfig: minInstances: 0 maxInstances: 50 weight: 1 basicAlgorithm: cooldownPeriod: 2m yarnConfig: scaleUpFactor: 1.0 scaleDownFactor: 1.0 gracefulDecommissionTimeout: 1h
🎯 Autoscaling Benefits
  • Cost Optimization: Scale down during idle periods
  • Performance: Scale up for large workloads
  • Automatic: No manual intervention required
  • Granular Control: Configure scaling policies per use case
